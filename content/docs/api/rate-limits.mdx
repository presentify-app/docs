---
title: Rate Limits
description: API request limits and throttling behavior.
---

The Presentify REST API enforces rate limits to ensure fair usage and system stability.

## Limits

| Scope | Limit | Window |
|-------|-------|--------|
| **Per IP address** | 150 requests | 60 seconds |

Rate limiting is applied **before** authentication. Even invalid API keys count toward the limit.

## How it works

- Uses a **fixed-window** counter backed by Redis
- IP address detected from `X-Forwarded-For` header (via Azure Front Door), then `X-Real-IP`, then connection IP
- Counter resets every 60 seconds

## When you hit the limit

If you exceed 150 requests in a 60-second window, the API returns:

```json
{
  "error": {
    "code": "TOO_MANY_REQUESTS",
    "message": "Rate limit exceeded. Please try again later."
  }
}
```

**HTTP Status:** `429 Too Many Requests`

## Best practices

- **Cache responses** — Store results locally instead of re-fetching
- **Use webhooks** — For real-time updates, use [webhooks](/docs/integrations/webhooks) instead of polling
- **Batch requests** — Fetch date ranges instead of individual dates
- **Respect 429 responses** — Implement exponential backoff when rate limited
- **Monitor usage** — Track request counts to stay within limits

## Redis unavailability

If the rate limiter's Redis instance is unavailable, the API defaults to **fail-closed** behavior — requests are denied until Redis recovers. This prevents uncontrolled load during infrastructure issues.
