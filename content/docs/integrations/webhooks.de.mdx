Webhooks senden HTTP-POST-Anfragen an deinen Server, wenn Events in Presentify passieren. Nutze sie zur Integration mit externen Systemen wie Slack, HR-Tools oder eigenen Dashboards.

> Webhooks erfordern einen **Enterprise**-Plan.

## Einen Webhook-Endpoint erstellen

1. Gehe zu **Einstellungen** > **Webhooks**
2. Klicke auf **Endpoint hinzufügen**
3. Konfiguriere:
   - **URL** (erforderlich) — Dein HTTPS-Endpoint
   - **Beschreibung** (optional) — Max 200 Zeichen
   - **Events** (erforderlich) — Wähle, welche Events du empfangen möchtest
4. Klicke auf **Speichern**

Das Webhook-**Secret** wird einmalig nach der Erstellung angezeigt. Kopiere es sofort — du brauchst es für die Signaturverifikation.

Du kannst bis zu **5 Webhook-Endpoints** pro Workspace erstellen.

|-------|-------------|
| `presence.created` | Ein Mitglied hat seinen Anwesenheitsstatus gesetzt |
| `presence.updated` | Ein Mitglied hat seinen Anwesenheitsstatus geändert |
| `presence.deleted` | Ein Mitglied hat seinen Anwesenheitsstatus entfernt |
| `desk_booking.created` | Ein Schreibtisch wurde gebucht |
| `desk_booking.cancelled` | Eine Schreibtischbuchung wurde storniert |

## Payload-Format

Jede Webhook-Zustellung verpackt die Event-Daten in ein Standard-Envelope:

```json
{
  "id": "delivery-uuid",
  "type": "desk_booking.created",
  "occurred_at": "2026-03-20T14:30:00.000Z",
  "data": {
    "booking_id": "...",
    "member_id": "...",
    "desk_id": "...",
    "date": "2026-03-21",
    "status": "confirmed"
  }
}
```

| Feld | Typ | Beschreibung |
|------|-----|-------------|
| `id` | `string` | Eindeutige Delivery-ID |
| `type` | `string` | Event-Typ (z.B. `desk_booking.created`) |
| `occurred_at` | `string` | ISO-8601-Zeitstempel des Events |
| `data` | `object` | Event-spezifischer Payload (siehe [Event-Typ-Details](#event-typ-details)) |

## HTTP-Header

| Header | Beschreibung |
|--------|-------------|
| `X-Presentify-Signature` | HMAC-SHA256 Hex-Digest des Payloads |
| `X-Presentify-Event` | Event-Typ (z.B. `desk_booking.created`) |
| `X-Presentify-Delivery` | Eindeutige Delivery-ID |
| `X-Presentify-Retry` | Retry-Versuchsnummer (nur bei Wiederholungen) |
| `Content-Type` | `application/json` |

## Signaturverifikation

Jede Webhook-Anfrage wird mit dem Webhook-Secret deines Workspaces per HMAC-SHA256 signiert. Verifiziere den `X-Presentify-Signature`-Header, um die Authentizität der Anfrage sicherzustellen.

**Node.js:**

```javascript
const crypto = require("crypto");

function verifySignature(secret, payload, signature) {
  const computed = crypto
    .createHmac("sha256", secret)
    .update(payload, "utf8")
    .digest("hex");
  return computed === signature;
}

// Verwendung in Express:
app.post("/webhook", (req, res) => {
  const sig = req.headers["x-presentify-signature"];
  if (!verifySignature(WEBHOOK_SECRET, req.rawBody, sig)) {
    return res.status(401).send("Invalid signature");
  }
  // Event verarbeiten...
  res.status(200).send("OK");
});
```

**Python:**

```python
import hmac
import hashlib

def verify_signature(secret: str, payload: str, signature: str) -> bool:
    computed = hmac.new(
        secret.encode("utf-8"),
        payload.encode("utf-8"),
        hashlib.sha256,
    ).hexdigest()
    return hmac.compare_digest(computed, signature)

# Verwendung in Flask:
@app.route("/webhook", methods=["POST"])
def webhook():
    sig = request.headers.get("X-Presentify-Signature")
    if not verify_signature(WEBHOOK_SECRET, request.data.decode(), sig):
        return "Invalid signature", 401
    # Event verarbeiten...
    return "OK", 200
```

## Retry-Verhalten

Wenn dein Endpoint einen Nicht-2xx-Statuscode zurückgibt oder einen Timeout hat (10 Sekunden), wiederholt Presentify die Zustellung:

| Versuch | Verzögerung |
|---------|------------|
| 1. Retry | 1 Minute |
| 2. Retry | 5 Minuten |
| 3. Retry | 30 Minuten |
| 4. Retry | 2 Stunden |
| 5. Retry | 12 Stunden |

Nach 5 fehlgeschlagenen Retries wird die Zustellung als fehlgeschlagen markiert. Jeder Versuch wird im Zustellungsprotokoll protokolliert.

## Test-Zustellung

Klicke auf **Testen** bei einem Webhook-Endpoint, um einen Test-Payload zu senden. Dies verifiziert, dass dein Endpoint erreichbar ist und korrekt antwortet.

## Zustellungsprotokoll

Jeder Endpoint zeigt aktuelle Zustellungen mit:

- Event-Typ und Delivery-ID
- Antwort-Statuscode
- Erfolgs-/Fehlerstatus
- Versuchsanzahl
- Zeitstempel

Nutze das Zustellungsprotokoll zum Debuggen von Integrationsproblemen.

## Best Practices

- **Schnell antworten** — Antworte sofort mit `200 OK` und verarbeite das Event dann asynchron.
- **Idempotenz sicherstellen** — Nutze den `X-Presentify-Delivery`-Header zur Deduplizierung. Du kannst dasselbe Event mehrfach erhalten.
- **Signatur immer verifizieren** — Verifiziere die Signatur bevor du den Payload verarbeitest, um die Authentizität sicherzustellen.
